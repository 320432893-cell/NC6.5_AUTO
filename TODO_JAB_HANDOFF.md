## 保留坑点

- JAB 不能由 WSL/Linux Python 直接控制 NC，实际操作必须用 Windows Python。
- JAB path 和 hwnd 不稳定，不要长期硬编码。
- 收款查询结果区的 `SunAwtCanvas` 标题为空，多个 Java Canvas 可能同时存在；只按 class/path 找控件有串窗风险，分页相关动作必须带 `scope_hwnd`。
- NC 结果表刷新慢时不要连续压命令；必须等分页 label 和结果表摘要稳定后，再读表/翻页/改每页条数。
- 指定列读取是性能优化，不是稳定性根因；真正要守的是“查询结果稳定前不读表、不翻页、不改分页”。
- 稳定条件看两个信号：分页 label 不变，候选结果表 row/col 摘要不变。
- 坐标点击、bounds 中心点点击、截图找坐标在 NC 自动化里彻底禁用；JAB 动作接口找不到控件时，必须继续探测 path/窗口/状态，不能做坐标回退。
- `ok=True` 只代表 JAB 动作返回成功，业务上必须做后置状态验证。
- 控件探索工具本身也可能截断真相；`tools/jab_probe.py --inspect-path` 曾经固定只展开 1 层，导致右侧查询条件区被误判为空。遇到“控件不存在/为空”时，先核查 `--depth`、`--max-children`、窗口 title/class、visible/showing 过滤是否真的生效。
- `--inspect-path` 输出里的 `path=0...` 是相对目标节点的路径，不是完整窗口路径；写入配置前必须用完整 path 再只读验证一次。
- 主查询入口默认走 F3；用 JAB action 点主界面查询按钮曾触发 Access Bridge 不稳定。
- 查询窗口内部 `正式单据` / `确定` 可走 JAB AccessibleAction，日期框走 `setTextContents`。
- `目的业务日期` 是 `介于`，限定当天必须起止两个日期框都填同一天。
- `目的业务日期` 只在选择 `正式单据` 后出现；未选择前的 `.2` 行是 `生效日期`，不能当作凭证查询日期。当前目的业务日期起止 path 是 `.11.1.0.0` / `.11.1.2.0`。
- `setTextContents` 对日期框有效，但 JAB 文本读取可能返回空，不能把读回空直接判定为写入失败。
- 收款单查询条件里的 `收款财务组织` 和 `单据日期` 也存在同样现象：`setTextContents` 成功后 JAB 读回可能为空，以界面状态或后续查询结果验证。
- 收款单表头【客户】不能沿用上面这个软口径。客户为空会影响保存业务正确性，所以必须作为保存前硬门槛。不要为了跑通保存测试而把客户字段软通过。
- 现场操作 NC GUI 前必须先说清楚准备做什么；不要静默改方案，不要点未知按钮，不要在财务组织输入框里随手按 Enter。
- 隐藏或 `visible=False` 的 `SunAwtDialog` 查询窗口可能是残留，不能作为可操作窗口依据。
- 查询窗口右侧筛选区和左侧会计科目树必须隔离，自动化不要碰左侧会计科目。
- NC 查询条件区视觉换行不等于 JAB 结构，定位要结合 label、role、row 容器、bounds 和后置状态验证。
- 严重坑点：无标题小 `SunAwtWindow` 不一定是残留。`新增 -> 自制/应收单` 菜单、参照/下拉也可能是这种窗口。旧逻辑禁用/隐藏可见小窗后，会造成菜单贴图、背后残影粘住、按钮点了没反应，属于阻塞级故障。
- `hide_blank_awt_windows()` 只能作为窄范围残留清理入口。打开菜单、参照、下拉的动作必须用 `cleanup_blank_awt=False`，然后递归当前 popup 控件树；业务 popup 打开期间禁止泛清。
- `新增 -> 自制` 要按 popup 生命周期清理：点击 `新增` 前快照窗口，点击后记录本次新增出来、包含 `自制/应收单` 菜单项的可见 `SunAwtWindow` hwnd；点完 `自制` 后 hide/close 这个 hwnd，并显式清理所有无标题小型 `SunAwtWindow` 残留。残留清理不按 visible 区分，不可见小窗也要清。
- JAB action 默认不自动 cleanup；`tools/jab_action_once.py` 也只有显式 `--cleanup-blank-awt` 才清。手工清残留用 `tools/close_awt_popup_residue.py --all-small`；旧 `--all-disabled-small` 是兼容别名，行为相同。
- bounds 点击在底层已拒绝，收款分页也取消了 bounds fallback；配置校验禁止 `next_bounds_timeout` 和 `open_query.click_mode=bounds`。找不到 action 时继续递归控件树。
- 发现截图样残留时，先确认它不是当前业务 popup。若当前 popup 已完成选择，直接用显式残留清理处理无标题小型 `SunAwtWindow`，包括不可见残留；不要在业务 popup 打开期间自动 cleanup。
- 新严重坑点：`Alt+F`、账号输入和 Enter 如果用全局键盘发送，且 `使用权参照` 没有成为前台窗口，输入会落到用户当前聊天框或编辑器。后续所有全局键盘输入必须先确认 `GetForegroundWindow()` 等于目标 hwnd；不相等时只能返回错误，不能尝试输入。
- 不要把“前台化、输入、等待结果”合并成一个长命令。先做只读/短前台检查，再由用户确认后执行输入。否则用户看不到当前到底是在等待、输入还是卡住。
- `tools/archive/window_overlay_probe.py --all` 输出极长，不适合作为高频现场操作；需要查 `使用权参照` hwnd 时应写窄范围过滤工具，避免信息噪声。`tools/archive/` 只作历史复盘证据，不是正式入口目录。
- 收款单 `自制录入态` 用上方 `保存`、`暂存`、`取消` 三个按钮同时存在来识别；`自制空白态` 如需单独判断，再叠加关键必填字段为空，第一版可先用 `财务组织` 输入框为空作为 ready-for-entry 判据。
- Excel/WPS 打开文件时可能导致写回失败；当前会包装成 `ExcelLockedError`，但 NC 已完成的业务动作无法自动撤销。
- 多行批量保存时，凭证号顺序可能不等于 Excel 顺序。严格顺序主线使用 `single`。
