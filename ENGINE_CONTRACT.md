# NC 引擎契约 (Engine Contract)

> 本文件是 NC6.5_AUTO 作为「引擎子进程」被外部 GUI 外壳(finance_tool 桌面壳)调用的**唯一接口真相源(SSOT)**。
> 两边照此各自动工:GUI 按本文件调命令、读进度、解结果、发停止;NC 按本文件保证命令的输入面、进度产物、结果产物、退出码、停止语义。
> 任何接口变更必须先改本文件再改代码。状态标注:`[现状符合]` 已满足契约;`[需改:小/中]` 待适配(对应《适配清单》六项),括注是否碰稳定业务逻辑。
>
> 适用拓扑:GUI 当主壳,NC 当引擎子进程。GUI 启动 NC 命令 → 轮询状态文件看进度 → 读 stdout 结构化结果 → 必要时落停止标志叫停。NC 始终独占自己的进程、急停、Excel 归属与断点续跑;**核心业务逻辑(四段式流程/匹配/页面状态机/控件定位/收款单逻辑)不在本次适配范围内,均不改动。**

---

## 0. 调用模型

```
GUI 外壳 ──启动子进程(命令 + 参数)──▶ NC 引擎
   ▲  ▲                                   │
   │  └──轮询 run_state.json(实时进度)──┘
   │                                       │
   └──读 stdout 末尾的结构化结果 JSON ◀────┘
   │
   └──落「停止标志文件」──▶ NC 下一个安全检查点停下,写 aborted,可续跑
```

四条通道,逐一在 §1 定义:**参数(入)/ 进度(状态文件)/ 结果(stdout JSON)/ 停止(标志文件)**。

## 1. 通用协议(所有命令共用)

### 1.1 进度通道 —— 状态文件 `run_state.json`

GUI 实时进度靠**轮询这个文件**,不靠解析日志。文件由引擎原子写入(tmp→replace),GUI 只读。

**位置**:必须可被注入(见 §1.5),GUI 启动时指定其落盘目录,默认安装目录下的可写区。`[需改:小]`(不碰稳定逻辑)

**字段(目标契约)**:

| 字段 | 含义 | 现状 |
|---|---|---|
| `run_id` | 本次运行唯一标识 | `[现状符合]` |
| `command` | 命令名 | `[现状符合]` |
| `status` | running / success / aborted / cancelled / failed | `[现状符合]` |
| `stage` | 当前阶段名(字符串) | `[现状符合]` |
| `step_index` / `total_steps` | 当前第几步 / 共几步(GUI 算百分比用) | `[需改:小]`(新增字段,不碰逻辑) |
| `counts` | 业务计数(总数/已成/已失败等) | `[现状符合]` 但字段名需统一 `[需改:小]` |
| `current` | 当前正在处理的对象描述 | `[现状符合]` |
| `last_error` | 运行中最近一次错误(实时更新,非只在收尾写) | `[需改:小]`(现仅 finish 时写) |
| `events` | 最近 200 条带时间戳事件 | `[现状符合]` |
| `started_at`/`updated_at`/`ended_at`/`elapsed_s` | 时间戳 | `[现状符合]` |

**写入时机**:初始化、每次进阶段、每次更新计数、每个事件、收尾各写一次。`[现状符合]`(凭证线);**收款单线尚未接入状态记录** `[需改:中]`(接记录层,不碰收款单业务逻辑)。

### 1.2 结果通道 —— 命令收尾在 stdout 输出一段结构化结果 JSON

GUI 解析 stdout **最后一段** JSON 判定结果(命令可在前面打人类日志,但末尾必须是机器可读 JSON 块)。

**信封(目标契约)**:

```json
{
  "ok": true,
  "command": "<命令名>",
  "exit_code": 0,
  "summary": { "total": 10, "succeeded": 9, "failed": 1, "skipped": 0 },
  "items": [ { "ref": "<行号/键>", "outcome": "success|failed|skipped", "reason": "<失败原因或空>" } ],
  "error": { "category": "none|business|environment|aborted", "message": "" },
  "elapsed_s": 12.3,
  "resumable": { "can_resume": false, "resume_command": null }
}
```

**现状**:收款单完整流程入口已输出近似结构(`--json`,含逐行 step 明细 + 三档退出码),是标杆 `[现状符合]`;其余入口多为纯文本 `[需改:小]`(补结构化输出,不碰逻辑)。

### 1.3 退出码(统一约定)

| 码 | 含义 | GUI 如何反应 |
|---|---|---|
| 0 | 全部成功 | 绿色完成 |
| 1 | 部分成功 / 业务失败(有行没成) | 黄色,展示 `items` 里失败项 |
| 2 | 无可处理项 / 前置状态不满足 | 提示前置条件 |
| 3 | 被外部停止(见 §1.4) | 灰色已停止,提示可否续跑 |
| 4 | 环境错误 / 崩溃(Excel 被占用、DLL 缺失、NC 客户端不在等) | 红色,引导排查环境 |

**现状**:各入口退出码不统一(收款单完整流程已有 0/1/2 三档;其余多为 0 / 非0)。需收敛到上表 `[需改:小]`(不碰逻辑)。

### 1.4 停止通道 —— 外部标志文件

人将来站在 GUI 前、不在 NC 跟前,故现有「键盘空格/ESC 急停」不够用。

**协议**:GUI 叫停 → 在约定位置创建**停止标志文件** → 引擎在下一个安全检查点(已密布于所有耗时循环)发现它 → 停在可续跑的位置 → 写 `status=aborted`、退出码 3、结果信封标注 `resumable`。

**现状**:急停检查点已遍布所有热循环、且停后能靠续跑命令接着干 `[现状符合]`;缺的只是「检查点也看一下外部标志文件」这一个分支 `[需改:小,仅一处几行]`(不碰逻辑)。

> ⚠️ **本项是六项里唯一的安全敏感点**:必须保证停止只发生在「不会把单据停在保存到一半」的检查点上,续跑语义不被破坏。此项改完须在 Windows/活 NC 上专门验一次。

### 1.5 路径 / 配置注入(装到每台机器都能跑)

- 命令统一接受 `--config <路径>` 覆盖配置文件;配置加载已支持外传 `[现状符合]`。
- 配置里**绑机器的绝对路径**(凭证 Excel 路径、收款单 Excel 路径、JAB 原生组件 DLL 路径)必须改为「每台机器自填」或「GUI 启动时注入」 `[需改:小]`。
- 引擎兜底里**写死的 DLL 路径**删除,交还自动搜索 `[需改:小]`。
- 日志/状态/性能文件落盘位置:从「相对当前目录」改为「相对统一基目录」解析,否则 GUI 用别的目录启动会落错 `[需改:小]`,与 §1.6 打包基目录同源。

### 1.6 非交互 & 可冻结打包

- **非交互**:所有引擎命令必须能纯参数从头跑到尾,不等 stdin。危险动作的确认改由显式参数表达(不是运行中问)。
- **可冻结**:引入统一「基目录」解析(识别是否打包态),日志/状态/性能/入口找文件处统一走它 `[需改:中]`。JAB 原生组件是系统级、按系统路径找得到,**不用打进包**。
